SNS vs SQS

Aspect SNS SQS
Message Consumption Push Mechanism - SNS pushes messages to consumers Pull Mechanism - Consumers poll and pull messages from SQS
Use Case Fanout - Processing the same message in multiple ways Decoupling two applications and allowing parallel asynchronous processing
Persistence No persistence. Whichever consumer is present at the time of message arrival gets the message and the message is deleted. If no consumers are available then the message is lost after a few retries. Messages are persisted for a configurable duration (max 2 weeks). The consumer doesnâ€™t have to be up when messages are added to the queue.
Consumer Type Consumers might process messages in different ways All consumers are typically identical and process messages the same way (though in rare cases messages may be resent)
Sample Applications Image processing example: - User uploads image to S3 - SNS notifies 3 consumers: 1. Watermarks image 2. Creates thumbnail 3. Sends Thank You email All consumers receive the same message (image URL) and process in parallel. Jobs are submitted to SQS. Consumers process jobs asynchronously. If job frequency increases, more consumers can be added for better throughput.

Fan-Out Intro

  • SNS is an AWS service that coordinates and manages the delivery of messages to subscribing endpoints.
  • You can create an SNS topic (logical access point / communication channel) and publish messages using AWS CLI, SDK, or Console.
  • The SNS topic forwards messages to its subscribers (e.g., SQS queues).
  • You can subscribe multiple queues to the same topic to replicate a message over multiple queues. This is called fan-out.
