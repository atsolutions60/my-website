Read Replicas

Introduction

Description
A read replica is a copy of your main (or primary) database in Amazon RDS.
It stays in sync with the primary database, constantly updating itself with any new data.
But unlike the primary, a read replica is read-only, meaning it can only handle data reads, not changes like updates or deletions.

Why Use a Read Replica?

Benefits
When your application has many users, the primary database can become slow due to too many requests.
To solve this, read replicas are used to reduce the load on the primary database by handling all the "read" requests.
This way, the primary database can focus on handling "write" requests, making the whole system faster and more efficient.

How Does It Work?

ConceptExplanation
Data Syncing The read replica automatically receives data updates from the primary database. This means any changes made to the primary database are reflected in the read replica.
Read Traffic Distribution You can direct read requests from your application to the read replica instead of the primary database. This reduces load on the primary database.

Benefits of Using Read Replicas

Benefits
Improved Performance
High Scalability
Reliability and Backup
Special Note

Read replicas cannot automatically take over as the primary database if the primary fails.

Unlike Multi-AZ setups, which provide automatic failover, a read replica must be manually promoted to become the new primary if needed.

This means that read replicas are not a substitute for Multi-AZ configurations when high availability and automatic failover are required.

Same Region Vs Cross-Region Read Replicas

TypeDetails
Same-Region Read Replicas These replicas are created within the same AWS region as the primary database.

Useful for distributing read traffic locally and improving performance.

Choosing different AZs for your read replicas can improve fault tolerance.
Cross-Region Read Replicas These replicas are created in a different AWS region from the primary database.

Ideal for:
• Reducing latency for global users
• Disaster recovery (replica can be promoted)
• Data migration across regions
